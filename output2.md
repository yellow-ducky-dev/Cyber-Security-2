(XSS) occurs when an attacker injects malicious scripts into a web page. A reflected XSS attack happens when user input is taken from the URL and displayed directly in the HTML response without sanitization. The browser sees it as valid HTML/JavaScript and executes it.</p><h2><strong>How It Was Found</strong></h2><p>The /search route takes a query parameter q and renders it directly inside the HTML response:</p><p>// VULNERABLE CODE in server.js:</p><p>const query = req.query.q || '';</p><p>res.send(`Search results for: &lt;strong&gt;${query}&lt;/strong&gt;`);</p><p>⚠️  No sanitization! Whatever is in 'query' is rendered directly as HTML.</p><h2><strong>Attack Payload Used</strong></h2><p>URL: http://localhost:3001/search?q=&lt;script&gt;alert('XSS-ATTACK')&lt;/script&gt;</p><p>Also tested: &lt;img src=x onerror=alert('XSS')&gt;</p><p>Also tested: &lt;b style='color:red'&gt;INJECTED HTML&lt;/b&gt;</p><h2><strong>Result — Attack Successful</strong></h2><p>✅ Browser executed the injected script</p><p>✅ HTML injection confirmed — attacker can modify page content</p><p>✅ With HttpOnly cookie not set, document.cookie is accessible</p><h2><strong>Real-World Impact</strong></h2><ul><li>Steal session cookies → log in as victim without password</li><li>Redirect user to phishing page</li><li>Keylog user's passwords typed on the page</li><li>Deface the website for all visitors</li></ul><h2><strong>The Fix Applied in Secured App</strong></h2><p>// SECURE CODE — authController.js:</p><p>const validator = require('validator');</p><p>const query = validator.escape(req.query.q || '');</p><p>// &lt;script&gt; becomes &amp;lt;script&amp;gt; — harmless text</p><p>Also added Content Security Policy via Helmet: scriptSrc: ["'self'"] — blocks all inline scripts even if injected.</p><h1><strong>Attack 2 MD5 Password Cracking (V2)</strong></h1><h2><strong>What is the Vulnerability?</strong></h2><p>The application stores passwords as MD5 hashes. MD5 was designed for speed (checksums, file verification) NOT for passwords. Speed is the enemy of password security: attackers can compute billions of MD5 hashes per second.</p><h2><strong>Vulnerable Code Found</strong></h2><p>// VULNERABLE CODE — register route:</p><p>const md5 = require('md5');</p><p>users.push({ password: md5(password) });</p><p>⚠️  MD5 has no salt. The same password always produces the same hash. Rainbow tables pre-compute every common password.</p><h2><strong>Attack — Hash Cracking</strong></h2><p>The following MD5 hashes were extracted from the application's user database (via the IDOR vulnerability, see Attack 4):</p><p>admin password hash:   0192023a7bbd73250516f069df18b500</p><p>alice password hash:   28db548ac69921eca66afe7de34f67f5</p><p>bob password hash:     64a53a0a2b7e2e8e3c5e78d79be22c87</p><h2><strong>Cracking Process</strong></h2><p>Step 1: Copy hash: 0192023a7bbd73250516f069df18b500</p><p>Step 2: Go to https://crackstation.net</p><p>Step 3: Paste hash → Click 'Crack Hashes'</p><p>Step 4: Result: 0192023a7bbd73250516f069df18b500 = admin123</p><p>✅ Hash cracked in &lt; 1 second using free online tool</p><p>✅ admin:admin123 confirmed — full account takeover</p><h2><strong>The Fix Applied in Secured App</strong></h2><p>// SECURE CODE — authController.js:</p><p>const bcrypt = require('bcrypt');</p><p>const hashedPassword = await bcrypt.hash(password, 10);</p><p>// Same password → different hash every time (auto-salted)</p><p>// Cost factor 10 = ~100ms to compute — brute force impractical</p><p>bcrypt also auto-generates a unique salt per password, so two users with the same password have completely different hashes.</p><h1><strong>Attack 3 Username Enumeration via Verbose Errors (V3)</strong></h1><h2><strong>What is the Vulnerability?</strong></h2><p>The login page returns different error messages depending on whether the username exists. This allows an attacker to enumerate (discover) valid usernames without knowing the password.</p><h2><strong>Test Commands &amp; Results</strong></h2><p>Test 1 — Non-existent username:</p><p>POST /api/login {"username":"hacker123","password":"test"}</p><p>RESPONSE: {"error":"Username not found: hacker123"}</p><p>Test 2 — Real username, wrong password:</p><p>POST /api/login {"username":"admin","password":"wrongpass"}</p><p>RESPONSE: {"error":"Wrong password for: admin"}</p><p>⚠️  Two different error messages = attacker knows 'admin' is a valid username. They can now target it with brute force.</p><h2><strong>Real-World Attack Chain</strong></h2><ul><li>Step 1: Try 1000 common usernames — collect ones that return 'Wrong password'</li><li>Step 2: Now brute-force only the confirmed valid usernames</li><li>Step 3: Far fewer attempts needed → faster crack</li></ul><h2><strong>The Fix Applied in Secured App</strong></h2><p>// SECURE CODE — authController.js:</p><p>return res.status(401).json({ error: "Invalid credentials." });</p><p>// Same message whether username OR password is wrong</p><p>Generic message reveals nothing to the attacker.</p><h1><strong>Attack 4 — IDOR: Insecure Direct Object Reference (V4)</strong></h1><h2><strong>What is IDOR?</strong></h2><p>IDOR occurs when an application uses a user-controllable input to access objects (data records) directly without checking if the requester is authorized. Simply changing a number in the URL gives access to someone else's data.</p><h2><strong>Vulnerable Code</strong></h2><p>// VULNERABLE CODE — server.js:</p><p>app.get('/api/user/:id', (req, res) =&gt; {</p><p>  const user = users.find(u =&gt; u.id == req.params.id);</p><p>  res.json(user); // No auth check! Returns everything including password hash</p><p>});</p><h2><strong>Attack — Accessing All Users' Data</strong></h2><p>Simply changing the ID number in the URL exposes every user's data:</p><p>GET http://localhost:3001/api/user/1</p><p>{"id":1,"username":"admin","password":"0192023a7bbd73250516f069df18b500","email":"admin@site.com","role":"admin"}</p><p>GET http://localhost:3001/api/user/2</p><p>{"id":2,"username":"alice","password":"28db548ac69921eca66afe7de34f67f5","email":"alice@site.com","role":"user"}</p><p>⚠️  No login required. Any anonymous user can dump all accounts including the admin's password hash and email.</p><h2><strong>The Fix</strong></h2><p>// SECURE: Check if requester is the owner OR an admin</p><p>app.get('/api/user/:id', authenticateToken, (req, res) =&gt; {</p><p>  if (req.user.id !== req.params.id &amp;&amp; req.user.role !== 'admin')</p><p>    return res.status(403).json({ error: 'Forbidden' });</p><p>  // Also never return password hash in response</p><p>});</p><h1><strong>Attack 5 — Brute Force via No Rate Limiting (V5)</strong></h1><h2><strong>What is Brute Force?</strong></h2><p>When there is no rate limiting, an attacker can automate thousands of login attempts per second until the correct password is found. Combined with the username enumeration (Attack 3), this is a complete account takeover method.</p><h2><strong>Test — PowerShell Brute Force Loop</strong></h2><p>The following loop was run in PowerShell against the vulnerable app:</p><p>for ($i=1; $i -le 11; $i++) {</p><p>  Invoke-WebRequest -Uri 'http://localhost:3001/api/auth/login'</p><p>    -Method POST -ContentType 'application/json'</p><p>    -Body '{"username":"test","password":"test"}'</p><p>}</p><h2><strong>Results No Blocking</strong></h2><p>Attempt 1:  {"success":false,"msg":"fail"}</p><p>Attempt 2:  {"success":false,"msg":"fail"}</p><p>Attempt 3:  {"success":false,"msg":"fail"}</p><p>Attempt 4:  {"success":false,"msg":"fail"}</p><p>Attempt 5:  {"success":false,"msg":"fail"}</p><p>Attempt 6 (correct password): {"success":true,"msg":"CRACKED! user=admin"}</p><p>⚠️  Zero rate limiting = unlimited attempts. A real attacker uses wordlists with millions of common passwords and cracks most accounts within minutes.</p><h2><strong>Comparison — Secured App Rate Limiter in Action</strong></h2><p>On the secured app the same loop produced:</p><p>Attempts 1-9:  {"error":"Invalid credentials."}</p><p>Attempt 10+:  HTTP 429 — {"error":"Too many login attempts. Try again in 15 minutes."}</p><p>This was demonstrated live in the terminal during testing — confirmed working.</p><h2><strong>The Fix Applied in Secured App</strong></h2><p>const rateLimit = require('express-rate-limit');</p><p>const loginLimiter = rateLimit({</p><p>  windowMs: 15 * 60 * 1000, // 15 minutes</p><p>  max: 10,                   // 10 attempts max</p><p>  message: { error: 'Too many login attempts.' }</p><p>});</p><p>app.use('/api/auth', loginLimiter);</p><h1><strong>Attack 6 Missing Security Headers (V6)</strong></h1><h2><strong>What Are Security Headers?</strong></h2><p>HTTP response headers can instruct the browser to enforce security policies. Without them, the browser applies no restrictions — enabling clickjacking, MIME sniffing, and XSS attacks.</p><h2><strong>Test Checking Headers</strong></h2><p>Checking the vulnerable app's response headers using curl:</p><p>curl -I http://localhost:3001/</p><h2><strong>Vulnerable App Headers — Missing Everything</strong></h2><p>HTTP/1.1 200 OK</p><p>X-Powered-By: Express          ← EXPOSES TECHNOLOGY STACK</p><p>Content-Type: text/html        ← No security headers present</p><p>(No X-Frame-Options)           ← Clickjacking possible</p><p>(No Content-Security-Policy)   ← XSS unrestricted</p><p>(No X-Content-Type-Options)    ← MIME sniffing possible</p><p>(No Strict-Transport-Security) ← HTTPS not enforced</p><h2><strong>Secured App Headers — Helmet.js Active</strong></h2><p>Content-Security-Policy: default-src 'self'  ← Only own scripts allowed</p><p>X-Frame-Options: SAMEORIGIN               ← Clickjacking blocked</p><p>X-Content-Type-Options: nosniff           ← MIME sniffing blocked</p><p>X-Powered-By: (removed)                   ← Stack hidden from attacker</p><h2><strong>The Fix Applied</strong></h2><p>const helmet = require('helmet');</p><p>app.use(helmet());  // Adds 12 security headers in one line</p><h1><strong>Week 3 Security Logging Evidence</strong></h1><h2><strong>Winston Log Output (security.log)</strong></h2><p>After all attacks were run, the security log captured every event with timestamps:</p><p>[2026-06-09T14:23:01] INFO: POST /api/auth/login - IP: ::1</p><p>[2026-06-09T14:23:01] INFO: POST /api/auth/login - IP: ::1</p><p>[2026-06-09T14:23:02] WARN: Rate limit hit: ::1 on /api/auth/login</p><p>[2026-06-09T14:23:02] INFO: POST /api/auth/login - IP: ::1</p><p>[2026-06-09T14:23:03] WARN: Rate limit hit: ::1 on /api/auth/login</p><p>This audit trail is critical in real-world security — it provides forensic evidence of attacks, detects ongoing threats, and feeds into SIEM systems.</p><h1><strong>Conclusion</strong></h1><p>All 8 vulnerabilities were successfully identified and exploited in the vulnerable demo application.
